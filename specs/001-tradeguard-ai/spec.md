## User Scenarios & Testing *(mandatory)*

> 핵심 철학: **"시장을 맞추는 것이 아니라, 사용자가 무너지는 순간을 먼저 감지하는 것."** 모든 사용자 시나리오는 이 철학에 부합해야 한다.

### User Story 1 — 거래 기록 업로드 및 행동 패턴 분석 (Priority: P1)

해외선물 트레이더가 자신의 과거 거래 내역(CSV)을 업로드하면, 시스템은 단순한 손익 통계가 아니라 **행동 패턴**(손절 지연, 복구매매, 연속 손실 후 진입, 시간대별 승률, 변동성 구간 성과)을 정량 점수로 변환해 보여준다.

**Why this priority**: 이 시스템의 모든 가치는 **사용자 자신의 매매 기록**으로부터 출발한다. 거래 업로드와 규칙 기반 행동 분석이 없으면 다른 기능(회고·시장 매칭·진입 위험도)은 작동할 수 없으며, 이 단독만으로도 "내 매매 약점을 객관적으로 본다"는 핵심 가치를 전달하는 MVP가 성립한다.

**Independent Test**: 사용자가 회원가입 후 표준 CSV 파일을 업로드하면, 시스템이 각 거래를 저장하고 손절 지연 점수·복구매매 점수·연속 손실 streak·시간대별 승률을 계산해 대시보드에 표시한다. 외부 시장 API나 GPT 호출 없이도 완전한 흐름이 동작한다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 비어 있는 대시보드를 보고 있을 때, **When** 사용자가 표준 컬럼(종목·진입시간·청산시간·방향·진입가·청산가·손익·계약수)을 갖춘 CSV를 업로드하면, **Then** 모든 거래가 DB에 저장되고 대시보드에 거래 목록과 기본 통계(승률, 누적 손익, 평균 보유시간)가 표시된다.
2. **Given** 거래가 업로드된 사용자가 분석 페이지로 이동했을 때, **When** "행동 분석" 결과를 요청하면, **Then** 손절 지연 점수, 복구매매 점수(연속 손실 후 10분 이내 재진입 비율 기반), 연속 손실 streak 최댓값, 시간대별 승률 차트가 표시된다.
3. **Given** 컬럼이 누락되었거나 손익 합산이 맞지 않는 CSV를 업로드할 때, **When** 시스템이 검증에 실패하면, **Then** 사용자에게 어떤 행·어떤 컬럼이 문제인지 명확히 알려주고 부분 저장 없이 거부한다.

### User Story 2 — AI 회고 리포트 (Priority: P2)

업로드된 거래 중 특정 거래(또는 기간)를 선택하면, 시스템은 사용자 자신의 행동 패턴과 결합한 **냉정한 회고 텍스트**를 생성한다. "위로"가 아니라 "분석"의 톤이어야 한다.

**Why this priority**: P1의 정량 점수만으로는 사용자가 자신의 매매 의미를 체감하기 어렵다. 자연어 회고는 정량 점수를 "행동 서사"로 번역해 자기 인식 단계를 한 단계 끌어올린다. P1이 있어야 의미가 있으므로 P2.

**Independent Test**: P1 완료 상태에서 특정 손실 거래를 클릭하면, "이번 손실은 연속 손실 2건 직후 평소 평균 보유시간의 3배를 들고 있었으며, 손절 기준을 후행적으로 수정한 패턴과 유사합니다" 같은 회고가 생성된다. 시장 데이터 없이도 동작해야 한다.

**Acceptance Scenarios**:

1. **Given** P1 분석이 완료된 거래 목록에서, **When** 사용자가 특정 거래의 "회고 생성"을 클릭하면, **Then** 해당 거래의 정량 지표(보유시간, 손절 지연 점수, 직전 연속 손익)를 입력으로 한 회고 텍스트가 생성된다.
2. **Given** 한 주(또는 임의 기간) 범위를 선택했을 때, **When** "주간 회고"를 요청하면, **Then** 그 기간 동안의 행동 패턴 변화(예: 복구매매 빈도 증가)를 짚는 요약 회고가 생성된다.
3. **Given** 회고 결과가 표시된 상태에서, **When** 사용자가 회고 톤을 확인하면, **Then** "괜찮아요" 류의 위로 표현이 등장하지 않고, 패턴·확률·과거 유사 거래를 근거로 한 분석 문장만 포함된다.

### User Story 3 — 시장 컨텍스트 매칭 (Priority: P3)

각 거래의 진입 시각을 기준으로 당시의 시장 상태(VIX, DXY, ATR, 거래량, CPI/FOMC 일정 등) 스냅샷을 연결한다. 이를 통해 회고와 행동 분석이 "변동성 이벤트 직전/직후" 같은 맥락을 포함할 수 있다.

**Why this priority**: 시장 컨텍스트가 더해지면 "왜 그 시점에 그런 행동을 했는가"라는 인과 해석이 가능해진다. 하지만 외부 API 의존도가 높아 신뢰성/비용 리스크가 따르므로 P1·P2 이후에 추가한다.

**Independent Test**: 과거 거래 한 건을 선택하면, 진입 시각의 VIX·DXY·ATR 값과 향후 1시간 내 예정된 경제 이벤트가 표시된다. 시장 데이터가 없는 시간대(데이터 공백)에 대해서는 "데이터 없음"으로 명시한다.

**Acceptance Scenarios**:

1. **Given** P1으로 저장된 거래가 있을 때, **When** 일괄 "시장 컨텍스트 채우기"를 실행하면, **Then** 각 거래에 대응하는 시장 스냅샷(VIX/DXY/ATR/거래량/이벤트 타입)이 저장되고 분석 페이지에 표시된다.
2. **Given** 시장 컨텍스트가 채워진 거래에서, **When** 사용자가 회고를 다시 생성하면, **Then** 회고 텍스트에 "CPI 발표 1시간 전" 같은 시장 상태 단서가 포함된다.
3. **Given** 무료 시장 데이터 API가 일시적으로 실패할 때, **When** 컨텍스트 채우기가 실행되면, **Then** 실패한 거래는 "컨텍스트 없음"으로 표시되고 사용자가 나중에 재시도할 수 있다.

### User Story 4 — 실시간 진입 위험도 (Priority: P4)

사용자가 "지금 NQ Long 진입해도 될까?" 같이 후보 거래의 종목·방향을 입력하면, 시스템은 (a) 현재 시장 상태, (b) 사용자 과거 유사 상황에서의 결과, (c) 현재 행동 리스크(직전 연속 손실·복구매매 신호), (d) 사전 멘탈 체크인 점수(Tilt Score), (e) Prop Firm 룰 위반 거리를 결합해 **위험도 점수**, **유사 과거 패턴**, **경고 메시지**를 출력한다.

**Why this priority**: 이 기능이 시스템의 가장 차별적인 가치 — "무너지는 순간을 먼저 감지" — 를 가장 직접적으로 구현한다. 그러나 P1~P3, P5, P6이 모두 정확해야 의미 있는 출력을 낼 수 있으므로 가장 마지막에 둔다.

**Independent Test**: P1~P3, P5, P6이 완료된 사용자가 "NQ Long, 1계약"을 입력하면, 위험도 점수(0~100), 과거 유사 시장 상태에서의 본인 승률, 현재 행동 리스크 경고, Tilt 상태(Green/Yellow/Red), Prop Firm 일일 손실 한도까지 남은 여유가 표시된다.

**Acceptance Scenarios**:

1. **Given** 충분한 거래 이력과 시장 컨텍스트가 있는 사용자가, **When** 후보 거래(종목·방향)를 입력하면, **Then** 0~100 위험도 점수와 그 점수의 근거(행동 신호·시장 신호·유사 과거 패턴·Tilt·Prop Firm 룰 여유)가 표시된다.
2. **Given** 직전 1시간 내 2건 이상의 손절이 있는 상태에서, **When** 사용자가 새 진입을 시도하면, **Then** 위험도가 임계치 이상으로 산출되고 명시적 경고("현재 행동은 과거 대손실 패턴과 유사합니다")가 표시된다.
3. **Given** Tilt 체크인이 Red 상태인 사용자가, **When** 진입 시도하면, **Then** 위험도가 자동으로 임계치 이상이 되고 경고가 강조된다.
4. **Given** 사용자가 시스템 경고를 무시하고 거래를 실제로 체결한 경우, **When** 해당 거래가 사후 업로드되면, **Then** "사전 경고가 있었음"이 회고에 자동 포함된다.

### User Story 5 — Prop Firm 컴플라이언스 (Priority: P3)

Topstep·Apex·FTMO 같은 펀딩 회사 계정으로 거래하는 사용자가 자신의 펀딩 룰(계정 잔고·일일 손실 한도·트레일링/정적 드로우다운·룰 종류)을 설정하면, 시스템은 매 거래마다 룰 위반까지 남은 거리를 실시간으로 표시하고, 위반 임박 시 경고한다.

**Why this priority**: 해외선물 트레이더 상당수가 펀딩 회사를 사용하며, "계정 폭파"는 가장 즉시적·물리적 리스크다. P1~P2가 있어야 거래 데이터 기반 룰 추적이 가능하므로 P3.

**Independent Test**: 사용자가 "Topstep 50K, EOD 트레일링 $2,000, 일일 손실 한도 $1,000" 프로필을 설정한 후, 거래를 업로드하거나 위험도 평가를 요청하면, "오늘 추가 허용 손실 $420, 트레일링 드로우다운까지 $1,580 여유" 형태의 룰 여유 패널이 표시된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 펀딩 회사·계정 사이즈·룰 종류를 입력했을 때, **When** 프로필을 저장하면, **Then** 시스템은 일일 손실 한도, 최대 드로우다운, 룰 타입(EOD/Intraday/Static)을 저장하고 대시보드에 룰 여유 패널을 표시한다.
2. **Given** 거래가 업로드되어 일일 손실이 한도의 80%에 도달했을 때, **When** 사용자가 대시보드 또는 위험도 평가를 열면, **Then** 명시적 경고("일일 손실 한도 80% 도달")가 표시된다.
3. **Given** 트레일링 드로우다운 잔여가 임계치(예: $500) 이하일 때, **When** 사용자가 새 거래 위험도를 평가하면, **Then** 위험도 점수에 Prop Firm 룰 신호가 큰 가중치로 반영된다.

### User Story 6 — 사전 진입 멘탈 체크 (Tilt Score) (Priority: P2)

사용자가 거래 세션을 시작할 때 30초 이내 짧은 체크인을 통해 자신의 멘탈 상태를 신호등(Green/Yellow/Red)으로 기록한다. 이 신호는 이후의 모든 위험도 평가에 입력으로 들어간다.

**Why this priority**: 사전 멘탈 체크는 사후 분석으로 잡히지 않는 "행동 리스크의 선행 지표"다. P1의 정량 패턴과 결합되면 위험도 정확도가 크게 올라간다. Edgewonk·TradesViz가 검증한 패턴.

**Independent Test**: 사용자가 거래 세션 시작 화면에서 (a) 수면 점수(1-10), (b) 스트레스 점수(1-10), (c) 직전 외부 사건(자유 텍스트)을 입력하면, 시스템이 Tilt Score(Green/Yellow/Red)를 산출하고 세션 동안 유지·표시한다.

**Acceptance Scenarios**:

1. **Given** 로그인 직후 또는 새 세션을 시작할 때, **When** 사용자가 30초 체크인을 완료하면, **Then** 시스템은 Tilt Score를 산출해 세션 헤더에 신호등으로 표시하고 해당 세션에 저장한다.
2. **Given** Tilt Score가 Yellow일 때, **When** 사용자가 새 진입 위험도를 평가하면, **Then** "사이즈 50% 감소 권고" 메시지가 권장사항으로 표시된다.
3. **Given** 사용자가 체크인을 건너뛰었을 때, **When** 위험도 평가를 요청하면, **Then** "Tilt 신호 없음 — 체크인 권장" 안내가 표시되되 위험도 평가는 다른 신호로 진행된다.

### Edge Cases

- 사용자가 동일 거래를 두 번 업로드(중복 CSV 행)할 때 — 시스템은 (종목, 진입시간, 청산시간, 진입가, 청산가) 조합을 기준으로 중복을 감지하고 추가 저장을 막아야 한다.
- 보유 중인(미청산) 포지션이 CSV에 포함된 경우 — `closed_at`/`exit_price`가 비어 있는 행은 별도로 표시하거나 명시적으로 제외 처리한다.
- 시간대(timezone) 충돌 — 사용자 로컬 시간과 시장 시간(미국 동부)이 다를 때, 분석은 UTC를 내부 기준으로 사용하되 표시는 사용자 로컬로 한다.
- 단 한 건의 거래만 업로드된 경우 — "행동 패턴" 같은 통계는 의미 없으므로 "분석을 위한 최소 거래 수에 도달하지 못함" 안내를 표시한다.
- CPI/FOMC 같은 이벤트가 거래 후에 발생한 경우 — 진입 시각 이후의 이벤트는 컨텍스트에서 제외하거나 "예정된 이벤트"로만 별도 표시한다.
- AI 회고 생성 API 호출 실패/타임아웃 — 회고는 "재시도" 상태로 두고 정량 점수는 그대로 노출한다.

## Requirements *(mandatory)*

### Functional Requirements

**거래 데이터 입력 (Trade Ingestion)**

- **FR-001**: 시스템은 인증된 사용자가 자신의 계정으로 CSV 파일을 업로드해 다수의 거래 레코드를 한 번에 등록할 수 있도록 해야 한다.
- **FR-002**: 시스템은 업로드된 CSV에서 종목·진입시간·청산시간·방향(Long/Short)·진입가·청산가·손익·계약수의 8개 표준 필드를 내부 정규 형식으로 요구한다. 단 사용자가 표준 8개 컬럼을 직접 정리할 필요는 없으며, 시스템은 (a) 미리 등록된 브로커 프리셋(이베스트·NinjaTrader·TradingView 등)으로 CSV를 자동 인식해 매핑하고, (b) 프리셋과 매칭되지 않는 경우 사용자가 원본 컬럼을 표준 8개에 드래그·드롭으로 매핑하는 UI를 제공해야 한다. 매핑이 완료되지 않은 상태에서는 저장을 거부한다.
- **FR-002a**: 사용자가 한 번 정의한 매핑은 "사용자 매핑 프리셋"으로 저장되어 다음 업로드에 자동 적용되어야 한다.
- **FR-003**: 시스템은 동일 사용자의 (종목·진입시간·청산시간·진입가·청산가) 조합이 일치하는 거래를 중복으로 간주하고 재저장을 방지해야 한다.
- **FR-004**: 시스템은 손익(`pnl`) 값이 (청산가 − 진입가) × 계약수의 방향성과 일치하는지 검증하고, 불일치 시 사용자에게 경고를 표시해야 한다.

**행동 분석 (Behavioral Analysis)**

- **FR-005**: 시스템은 거래별로 손절 지연 점수, 복구매매 점수, 확신 과다 점수를 0~100 범위로 계산해 저장해야 한다.
- **FR-006**: 시스템은 사용자별로 평균 손절 지연, 평균 복구매매 간격, 최대 연속 손실 streak, 야간 매매 비율, 확신 과다 점수를 집계해 행동 프로파일을 산출해야 한다.
- **FR-007**: 시스템은 시간대별 승률(예: 0~24시간 1시간 단위), 변동성 구간별 성과(낮은/보통/높은 ATR 구간) 차트를 제공해야 한다.
- **FR-008**: 모든 행동 점수는 계산식이 결정론적(deterministic)이어야 하며, 동일 입력에 대해 항상 동일한 결과를 내야 한다.

**시장 컨텍스트 (Market Context)**

- **FR-009**: 시스템은 각 거래의 진입 시각을 기준으로 시장 스냅샷(VIX, DXY, 거래량, ATR, 다가오는 경제 이벤트 타입)을 조회·저장해야 한다.
- **FR-010**: 시장 데이터가 일시적으로 조회되지 않은 경우, 해당 거래의 시장 컨텍스트는 빈 상태로 두고 사용자가 재시도할 수 있어야 한다.
- **FR-011**: 시장 컨텍스트는 거래의 메타 정보로 유지되며, 사용자 거래 데이터와 동일 수준의 보안(인증 사용자만 접근)으로 보호되어야 한다.

**회고 및 진입 위험도 (Retrospective & Live Risk)**

- **FR-012**: 시스템은 단일 거래 또는 임의 기간을 입력으로 AI 회고 텍스트를 생성하되, 입력에는 정량 점수와 시장 컨텍스트가 포함되어야 한다.
- **FR-012a**: 외부 LLM에 거래 데이터를 전송하기 전, 시스템은 PII에 해당하는 필드(계좌번호·실명·이메일·브로커 식별자)를 제거하거나 결정론적 토큰으로 치환해야 한다. 종목·방향·진입가/청산가·시각·계약수 등 분석에 필요한 메타정보는 전송될 수 있다.
- **FR-013**: AI 회고 톤은 "냉정한 분석" 톤이어야 하며, 위로/격려 표현(예: "괜찮아요")이 출력에 포함되지 않도록 프롬프트와 후처리 필터로 제어해야 한다.
- **FR-014**: 사용자는 후보 거래(종목·방향, 선택적으로 계약수)를 입력해 0~100 위험도 점수, 점수 근거(행동/시장/유사 패턴 비중), 과거 유사 상황에서의 본인 결과를 받을 수 있어야 한다.
- **FR-015**: 위험도 산출은 (a) 직전 N시간 손익 흐름, (b) 현재 시장 컨텍스트, (c) 과거 유사 컨텍스트에서의 본인 승률, (d) 현재 세션의 Tilt Score, (e) Prop Firm 룰 여유 5개 신호를 명시적으로 결합해야 하며, 각 신호의 기여 비중이 출력에 함께 표시되어야 한다.

**Prop Firm 컴플라이언스 (Prop Firm Compliance)**

- **FR-020**: 사용자는 Prop Firm 종류(Topstep/Apex/FTMO/기타), 계정 잔고, 일일 손실 한도, 드로우다운 종류(Static/EOD Trailing/Intraday Trailing) 및 한도 금액을 프로필로 등록할 수 있어야 한다.
- **FR-021**: 시스템은 거래가 업로드되거나 위험도 평가가 요청될 때마다, 등록된 Prop Firm 프로필을 기준으로 (a) 오늘 추가 허용 손실, (b) 트레일링/정적 드로우다운까지 남은 여유를 계산해 표시해야 한다.
- **FR-022**: 시스템은 일일 손실이 한도의 80% 이상이거나 드로우다운 여유가 사용자 정의 임계치 이하일 때 명시적 경고를 발생시켜야 한다.

**Tilt Score (사전 멘탈 체크)**

- **FR-023**: 시스템은 사용자가 거래 세션 시작 시 30초 이내에 입력 가능한 체크인 폼을 제공해야 한다(수면 1-10, 스트레스 1-10, 직전 외부 사건 자유 텍스트).
- **FR-024**: 시스템은 체크인 입력에서 Tilt Score(Green/Yellow/Red)를 결정론적 규칙으로 산출해 저장하고, 활성 세션 동안 헤더에 표시해야 한다.
- **FR-025**: 위험도 평가(FR-015)는 활성 세션의 Tilt Score를 가중치 입력으로 사용해야 하며, Red 상태에서는 위험도가 자동으로 임계치 이상이 되어야 한다.

**계정·접근 제어 (Account & Access)**

- **FR-016**: 시스템은 사용자별로 거래·분석·회고 데이터를 격리해야 하며, 어떤 사용자도 다른 사용자의 데이터에 접근할 수 없어야 한다.
- **FR-017**: 모든 거래 업로드·분석·회고·위험도 조회 요청은 인증된 세션에서만 허용되어야 한다.
- **FR-017a**: 사용자 인증은 Google OAuth와 Kakao OAuth 두 채널을 지원해야 한다. 비밀번호 기반 로컬 계정은 v1 범위 밖이다. 동일 사용자가 두 채널을 모두 연동해 어느 쪽으로든 로그인할 수 있어야 한다.

**감사 및 신뢰성 (Auditability & Reliability)**

- **FR-018**: 모든 AI 회고와 위험도 산출 결과는 입력 스냅샷(거래 ID, 시장 컨텍스트 ID, 점수)과 함께 저장되어, 동일 입력에 대한 재생산이 가능해야 한다.
- **FR-019**: 사용자가 자신의 모든 거래·분석·회고 데이터를 삭제할 수 있는 명시적 경로가 있어야 한다.

### Key Entities *(include if feature involves data)*

- **User (사용자)**: 시스템을 사용하는 개인 트레이더. 인증 정보, 표시 이름, 가입 시각을 가진다. 모든 다른 엔티티는 User에 귀속된다.
- **Trade (거래)**: 한 번의 진입~청산 사이클을 나타낸다. 종목, 방향, 진입가/청산가, 진입시각/청산시각, 손익, 계약수, 소유 User를 포함한다. 미청산 포지션은 청산 필드가 비어 있을 수 있다.
- **MarketSnapshot (시장 스냅샷)**: 특정 시각·종목에 대한 시장 상태. VIX, DXY, 거래량, ATR, 이벤트 타입(CPI, FOMC, 일반 등)을 포함한다. 한 Trade는 0 또는 1개의 MarketSnapshot에 연결된다.
- **Analysis (분석 결과)**: 한 Trade에 대한 정량 점수(위험도, 복구매매, 확신 과다, 손절 지연)와 AI 회고 텍스트, 생성 시각을 포함한다. Trade 1건당 여러 Analysis가 누적될 수 있다(시간 경과에 따른 재분석).
- **BehavioralProfile (행동 프로파일)**: User 단위의 집계 지표(평균 손절 지연, 평균 복구매매 간격, 최대 연속 손실 streak, 야간 매매 점수, 확신 과다 점수). User 1명에 1개.
- **RiskAssessment (진입 위험도 평가)**: 사용자가 후보 거래에 대해 요청한 한 번의 평가 결과. 입력(종목·방향·요청시각), 점수, 근거 요약, 사용한 BehavioralProfile/MarketSnapshot/TiltCheck/PropFirmProfile 스냅샷, 신호별 기여 비중, 생성 시각을 포함한다.
- **PropFirmProfile (펀딩 회사 프로필)**: 사용자가 등록한 펀딩 계정 설정. Prop Firm 종류, 계정 잔고, 일일 손실 한도, 드로우다운 종류(Static/EOD Trailing/Intraday Trailing) 및 한도 금액, 현재 누적 손익, 룰 여유를 포함한다. User 1명에 0~N개.
- **TradingSession (거래 세션)**: 사용자가 시작·종료하는 거래 시간 블록. 시작 시각, 종료 시각, 시작 시 입력한 TiltCheck를 가진다. 거래(Trade)와 위험도 평가(RiskAssessment)는 해당 세션에 귀속된다.
- **TiltCheck (멘탈 체크인)**: 세션 시작 시 입력한 멘탈 상태. 수면 점수(1-10), 스트레스 점수(1-10), 직전 외부 사건 자유 텍스트, 산출된 Tilt Score(Green/Yellow/Red), 입력 시각을 포함한다. TradingSession 1건에 1개.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 표준 CSV를 업로드한 뒤 **2분 이내**에 자신의 행동 점수와 시간대별 승률 차트를 처음으로 확인할 수 있다.
- **SC-002**: 동일한 거래 데이터를 업로드한 두 명의 사용자(또는 동일 사용자의 재업로드)에서, 정량 행동 점수는 **100% 동일**해야 한다(결정론적 계산 검증).
- **SC-003**: AI 회고 결과 중 위로/격려 표현(예: "괜찮아요", "다음에 잘하면 돼요")이 포함된 비율이 **2% 이하**여야 한다(샘플 100건 무작위 검수 기준).
- **SC-004**: 실시간 진입 위험도 평가 요청은 사용자 입력 후 **5초 이내**에 점수·근거·과거 유사 패턴을 반환한다.
- **SC-005**: 위험도 점수가 70 이상으로 산출된 거래 중, 사후 회고에서 "사전 경고가 있었음"으로 자동 표시되는 비율이 **90% 이상**이다(경고-회고 일관성).
- **SC-006**: 사용자는 "내 데이터 전체 삭제" 요청 후 **즉시(같은 세션 안에서)** 모든 거래·분석·회고·행동 프로파일이 시스템에서 제거된 것을 확인할 수 있다.
- **SC-007**: 인증되지 않은 요청이 거래·분석·회고·Prop Firm 프로필 데이터에 접근할 수 있는 경로는 **0건**이어야 한다(공개 엔드포인트로의 우발적 노출 방지).
- **SC-008**: Tilt Score가 Red인 세션에서 요청된 위험도 평가는 **100%** 임계치 이상의 점수를 반환해야 한다(FR-025 일관성).
- **SC-009**: Prop Firm 프로필이 등록된 사용자의 거래가 일일 손실 한도의 80%를 초과한 시점에서 발생한 위험도 평가는 **100%** 명시적 경고를 포함해야 한다(FR-022 일관성).

## Assumptions

- 본 시스템의 1차 배포 모델은 **단일 사용자 클라우드 SaaS**다(인증된 본인 1명만 사용). 따라서 다중 사용자 공유·소셜 기능은 v1 범위 밖이다. 다만 인증과 표준 데이터 격리 원칙은 v1부터 적용한다(향후 다중 사용자 전환을 고려한 방어적 설계).
- 사용자 인증은 **Google OAuth + Kakao OAuth** 두 채널로 한정한다. 비밀번호 기반 계정·이메일 매직 링크는 v1 범위 밖이다.
- 거래 데이터의 1차 입력 채널은 **CSV 업로드**이며, 시스템은 주요 브로커(이베스트·NinjaTrader·TradingView 등)의 프리셋 자동 인식과 사용자 매핑 UI 폴백을 함께 제공한다. 브로커 API 직결은 미래 확장이며 v1 범위가 아니다.
- AI 회고·위험도 설명은 **외부 LLM(Anthropic/OpenAI 등) API**를 호출해 생성한다. 전송 전에 PII(계좌번호·실명·이메일·브로커 식별자)를 제거하거나 결정론적 토큰으로 치환한다. 종목·수치·시각 등 분석에 필수적인 메타데이터는 전송된다. 로컬 LLM은 v1 범위 밖이다.
- 시장 데이터는 **무료 티어 데이터 소스**로 시작한다. 따라서 데이터 지연·결측은 정상 동작으로 간주하고, "데이터 없음" UX가 회피 경로가 아니라 명시적 상태다.
- 행동 점수의 1차 산출 방식은 **규칙 기반**이며, 머신러닝 모델은 v1 범위가 아니다. AI는 회고 생성과 위험 설명에 사용되고, 점수 자체를 결정하지 않는다.
- 사용자의 트레이딩 시장은 주로 **해외 선물(NQ, ES 등)** 이며, 시간대 기준은 UTC를 내부 저장 표준으로 한다. 표시는 사용자 로컬 시간대로 변환한다.
- 사용자가 거래를 실제로 체결하는 행위는 시스템 밖에서 일어난다. 시스템은 **사전 경고**와 **사후 회고**만 제공한다. 자동 주문/자동 거래 차단(Circuit Breaker, Kill Switch)은 v2로 연기하며 v1에서는 강한 경고 UX까지만 구현한다.
- v1 확장 범위로 **Prop Firm 컴플라이언스(P3)** 와 **Tilt Score 사전 멘탈 체크(P2)** 를 포함한다. R-Multiple/MAE/MFE/Expectancy 등 추가 산업 표준 메트릭, 거래 태깅, 트레이딩 플랜 vs 실행 차이 추적, 경제 캘린더 사전 차단, 세금 리포트는 v1.5~v2 백로그로 분류한다.
- AI 회고의 톤 일관성은 **프롬프트 + 후처리 필터** 조합으로 달성하며, 전수 검증이 아니라 SC-003의 샘플 검수로 모니터링한다.
- 사용자 1명의 일평균 거래량은 수십 건 수준, 누적 수년치 데이터는 수만 건 수준으로 가정한다. 대규모 동시 사용자(>1,000명) 부하는 v1 범위 밖이다.
